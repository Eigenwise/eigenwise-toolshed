# workspace-init

One command to set up a Claude Code workspace for a project, new or existing. It runs a short
interview, then writes a populated `.claude/`: a `settings.json` that enables the right plugins for
your stack, a tailored `live-rules.md`, a codebase map, and a short note on how the project is meant to
be structured. It wires all of that around the plugin-reload boundary and then verifies, in the same
session, that each piece actually fires.

The point: a fresh clone or a brand-new repo goes from "no Claude setup" to "Claude already knows the
stack, the conventions, and where things live" in one guided pass, and the same pass works whether the
project is a Rust service, a Svelte site, an ML repo, or a docs wiki.

## What it sets up

workspace-init is an **orchestrator**. It doesn't reinvent the pieces the toolshed already has, it
sequences them and fills in the glue:

- **`.claude/settings.json`** — registers the `eigenwise-toolshed` marketplace and enables the core
  plugins ([codebase-mapper](../codebase-mapper), [live-rules](../live-rules),
  [sidequest](../sidequest)), plus stack-appropriate extras it detects or asks about (a language
  server, a frontend or Cloudflare plugin, `context7` for live docs, and so on).
- **`.claude/live-rules.md`** — a starter set of craft rules (atomic commits, simple design, surgical
  changes, verify before "done") plus stack-specific rules scoped to the files they apply to, all in the
  [live-rules](../live-rules) format so you can edit them freely afterward.
- **A codebase map** under `.claude/.codebase-info/` — generated by the
  [codebase-mapper](../codebase-mapper) `map-codebase` skill (skipped for a project that isn't code,
  like a wiki or a notes vault).
- **Structure notes** — a short `.claude/.codebase-info/structure.md` (or a rule) capturing how the
  project is meant to be laid out, so a greenfield repo has intent written down before there's code.
- **`CLAUDE.md`** — delegated to the built-in `/init` when the project wants one.

Nothing here is stack-specific in the plugin itself. The logic is generic; the choices come from a
reference catalog it consults (and you can extend) plus what it finds in your repo.

## The reload boundary

Enabling plugins in `settings.json` only takes effect after Claude reloads them. workspace-init is built
around that: it does everything that doesn't need the plugins first (write `settings.json`, write
`live-rules.md`, write the structure notes), then pauses and asks you to run `/reload-plugins` (or
restart). Once the plugins are live, it continues: builds the map, brings up the sidequest board, and
checks that the live-rules and map hooks are actually injecting. You watch it work end to end rather
than trusting that the files were written correctly.

## The self-improvement loop

Every workspace it sets up gets a baked-in **self-improvement rule**: after you finish a chunk of work,
if you hit friction (fumbled the stack, re-derived something that should've been written down, tripped
over a missing convention), the workspace nudges you to fix the root cause. Refine a live rule, update
the map, tweak `CLAUDE.md`, or spin up a new skill with `skill-creator`. Small and incremental, so the
workspace you set up on day one keeps getting sharper the more you use it.

For a deeper pass, the bundled **`retro`** skill runs a structured reflection on demand: it reviews the
session for recurring friction and proposes concrete improvements to the rules, the map, `CLAUDE.md`, or
your skills, then applies the ones you approve.

## A note on `CLAUDE.md`

The other toolshed plugins are emphatic about never touching `CLAUDE.md`, because their hooks handle
injection and editing it would be redundant. workspace-init is the deliberate exception: it's the
one-time static setup, so it's the right place to seed `CLAUDE.md` (via `/init`), and the
self-improvement loop is explicitly allowed to update it. That's the one seam where editing `CLAUDE.md`
is correct.

## Install

```text
/plugin marketplace add Eigenwise/eigenwise-toolshed
/plugin install workspace-init@eigenwise-toolshed
```

Then reload plugins (or restart Claude Code). You don't have to install the other three yourself,
workspace-init enables them in the project it sets up.

## Use

In the project you want to set up:

```text
/init-workspace
```

or just ask: *"set up a Claude workspace here"*, *"init this project for Claude"*. The `init-workspace`
skill runs the interview, writes the files, walks you through the reload, and verifies everything. Commit
`.claude/` when it's done so your whole team shares the same setup.
